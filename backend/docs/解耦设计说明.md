# 发票识别系统 - 解耦设计说明

## 一、设计原则

本系统严格遵循以下架构原则：

1. **单一职责原则（SRP）**：每个模块只负责一个明确的功能
2. **开闭原则（OCP）**：对扩展开放，对修改关闭
3. **依赖倒置原则（DIP）**：依赖抽象而非具体实现
4. **关注点分离（SoC）**：不同关注点分散到不同模块
5. **显式优于隐式**：使用显式的枚举和类型，避免字符串硬编码

## 三、关键设计模式

### 1. 工厂模式（Factory Pattern）
**应用场景**：
- `PreprocessorFactory` - 根据文件类型创建预处理器
- `ProviderClientFactory` - 根据提供商和 API 风格创建客户端
- `get_request_builder()` - 根据 API 风格创建请求构建器
- `get_response_parser()` - 根据 API 风格创建响应解析器

**优势**：
- 隐藏对象创建细节
- 便于添加新的实现类
- 符合开闭原则

### 2. 策略模式（Strategy Pattern）
**应用场景**：
- `RequestBuilder` - 不同的请求构建策略
- `ResponseParser` - 不同的响应解析策略
- `FilePreprocessor` - 不同的文件预处理策略

**优势**：
- 算法可以独立于使用它的客户端变化
- 避免使用多重条件判断
- 提高代码的可测试性

### 3. 模板方法模式（Template Method Pattern）
**应用场景**：
- `AIProviderClient` - 定义识别流程的骨架
- `FilePreprocessor` - 定义预处理流程的骨架

**优势**：
- 提取公共代码到父类
- 子类只需实现特定步骤
- 符合开闭原则

### 4. 仓储模式（Repository Pattern）
**应用场景**：
- 所有 Repository 类

**优势**：
- 隔离数据访问逻辑
- 便于单元测试（可以 mock）
- 统一数据访问接口

### 5. 编排器模式（Orchestrator Pattern）
**应用场景**：
- `RecognitionOrchestrator` - 编排识别流程

**优势**：
- 集中管理复杂的业务流程
- 各个组件职责单一
- 流程清晰易懂

        output,
        media_type="application/octet-stream",
        headers={
            "Content-Disposition": f"attachment; filename=task_{task_id}.new"
        }
    )

## 五、可维护性保证

### 5.1 单一职责
每个模块只负责一个明确的功能，修改某个功能时只需要修改对应的模块，不会影响其他模块。

### 5.2 低耦合
模块之间通过抽象接口通信，不直接依赖具体实现，降低了模块间的耦合度。

### 5.3 高内聚
相关的功能集中在同一个模块中，提高了代码的内聚性。

### 5.4 易测试
每个模块都可以独立测试，通过 mock 可以隔离依赖。

### 5.5 易理解
清晰的分层和命名，使得代码结构一目了然，新人可以快速上手。

## 六、性能优化

### 6.1 并发控制
使用 `RateLimitController` 控制 AI 请求的并发数，避免超出 API 限制。

### 6.2 异步处理
使用 `async/await` 进行异步处理，提高 I/O 密集型操作的性能。

### 6.3 批量处理
在 PDF 识别中，可以并发处理多页，提高整体速度。

## 七、安全性

### 7.1 配置隔离
AI 配置（包括 API Key）存储在数据库中，不硬编码在代码中。

### 7.2 权限验证
所有操作都需要验证用户权限，确保用户只能访问自己的数据。

### 7.3 输入验证
使用 Pydantic 进行输入验证，防止非法数据。

## 八、总结

本系统通过严格的分层和解耦设计，实现了以下目标：

1. **高可维护性**：每个模块职责单一，修改某个功能时只需要修改对应的模块
2. **高可扩展性**：添加新的 AI 提供商、文件格式、发票类型、导出格式都非常简单
3. **高可测试性**：每个模块都可以独立测试，通过 mock 可以隔离依赖
4. **高可读性**：清晰的分层和命名，代码结构一目了然
5. **高性能**：使用异步处理和并发控制，提高系统性能
6. **高安全性**：配置隔离、权限验证、输入验证，确保系统安全

通过这种设计，系统可以轻松应对未来的需求变化，无需大规模重构。
